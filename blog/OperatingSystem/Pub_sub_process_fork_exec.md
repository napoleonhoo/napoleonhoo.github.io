# 操作系统：子进程的`fork`与 `exec` 

> 摘要：本文围绕 Unix/Linux 进程模型中的一条经典主线展开：先看有哪些常见的子进程创建方式，再理解 `exec` 为什么不是“创建新进程”，而是“用新程序替换当前进程”。在此基础上，进一步对比两种常见工程方案：**`fork()` 后子进程直接执行当前程序中的函数**，以及 **`fork()` 后通过 `exec()` 执行另一个程序**。文章重点分析它们在执行边界、状态继承、隔离性、启动成本和工程适用场景上的差异，帮助你建立对 `fork -> exec -> wait` 这一 Unix 经典模型的整体认知。

## 一、为什么这个问题重要？

在 Unix/Linux 世界里，很多系统能力都建立在“**创建进程**”和“**替换进程映像**”这两个动作之上。（*进程映像：process image，进程镜像*）

比如：

- Shell 执行一条命令
- Web Server 创建 worker 进程
- 守护进程拉起子任务
- 父进程把任务隔离到独立子进程中执行
- 应用在运行过程中调用外部工具（如 `ls`、`grep`、`ffmpeg`）

理解这些机制，核心离不开两组接口：

1. **创建子进程**：`fork()`、`vfork()`、`clone()`、`posix_spawn()`
2. **替换当前进程映像**：`exec*()` 函数家族

很多初学者会把“创建新进程”和“执行新程序”混为一谈，但这其实是两个不同动作：

- `fork()`：**复制出一个子进程**
- `exec()`：**把当前进程替换成另一个程序**

这也是 Unix 经典模型 **fork -> exec -> wait** 的基础。

## 二、子进程创建方式与 `exec` 家族简介

理解本文，先抓住两个动作：

- `fork()`：创建一个子进程
- `exec()`：用新程序替换当前进程映像

也就是说，`fork()` 解决的是“多出一个进程”，`exec()` 解决的是“这个进程接下来执行谁”。

### 1. 常见的子进程创建方式

Unix/Linux 下和子进程创建相关的接口，常见有这几类：

- `fork()`：最经典、最常用。父子进程从 `fork()` 返回后分别继续执行，现代系统通常借助 COW（写时复制）优化开销。
- `vfork()`：为“马上就要 `exec()`”的场景设计，约束较强，理解即可，日常开发通常优先用 `fork()` 或 `posix_spawn()`。
- `clone()`：Linux 更底层的构造接口，可精细控制资源共享，常见于线程库、容器和内核相关场景。
- `posix_spawn()`：更工程化的封装接口，适合把“创建进程 + 初始化 + 执行程序”作为一个整体来使用。

如果只是建立主干认知，优先抓住 `fork + exec` 即可。

### 2. `exec` 的本质

很多人会误以为 `exec` 是“再创建一个新进程”，其实不是。

> **`exec` 不创建新进程，而是让当前进程变成另一个程序。**

它成功后通常具有几个关键特征：

- PID 通常不变
- 原程序的代码、数据、堆、栈会被新程序替换
- 成功后不会返回
- 未设置 `FD_CLOEXEC` 的文件描述符默认会被继承

### 3. `exec` 家族

常见的 `exec` 变体包括：

- `execl`
- `execv`
- `execlp`
- `execvp`
- `execle`
- `execve`

根据后缀不同有：

- `l` = list，参数逐个列出
- `v` = vector，参数通过数组传入
- `p` = PATH，从环境变量 `PATH` 搜索程序
- `e` = env，显式传入环境变量

其中最核心的是：

```c
int execve(const char *pathname, char *const argv[], char *const envp[]);
```

它是最底层的接口，其他很多 `exec` 形式都可以看作是在它之上的易用封装。

### 4. 一个最常见的使用模型

Unix 里最经典的组合是：

1. 父进程 `fork()` 出子进程
2. 子进程 `exec()` 执行目标程序
3. 父进程 `wait()` / `waitpid()` 回收子进程

例如：

```c
pid_t pid = fork();
if (pid < 0) {
    perror("fork");
    return 1;
}

if (pid == 0) {
    execl("/bin/ls", "ls", "-l", NULL);
    perror("execl");
    _exit(1);
} else {
    waitpid(pid, NULL, 0);
}
```

这个模型几乎就是 shell、任务调度器、外部命令执行器等场景的基础。

## 三、`exec` 成功之后，进程里到底发生了什么？

这是理解 `exec` 的核心。

假设当前进程调用：

```c
execv("/bin/ls", args);
```

如果成功，那么：

### 1. 进程 PID 通常不变

还是同一个进程号，只是它执行的程序变了。

### 2. 代码段、数据段、堆、栈被新程序替换

原来程序的：

- 全局变量
- 堆内存
- 调用栈
- 函数逻辑

都不再延续。

### 3. 打开的文件描述符默认仍会保留

新程序虽然“换了代码”，但仍沿用那些没被关闭的文件描述符。除非这些文件描述符设置了 `FD_CLOEXEC`。

### 4. 大多数自定义信号处理行为会重置

很多信号相关状态在 `exec` 后会恢复默认行为。这也是为什么 `exec` 后的进程更像“一个干净的新程序映像”。

### 5. 成功后不会返回

如果你看到 `exec` 后面的代码还在运行，通常说明：

- `exec` 失败了
- 路径不对
- 程序不可执行
- 参数或环境有问题

## 四、为什么 Unix 经常是 `fork -> exec -> wait`？

这是非常经典的父子进程协作模型。

### 基本流程

1. 父进程 `fork()` 出一个子进程
2. 子进程调用 `exec()` 执行新程序
3. 父进程调用 `wait()` / `waitpid()` 等待子进程结束

示例：

```c
pid_t pid = fork();
if (pid < 0) {
    perror("fork");
    return 1;
}

if (pid == 0) {
    execl("/bin/ls", "ls", "-l", NULL);
    perror("execl");
    _exit(1);
} else {
    waitpid(pid, NULL, 0);
}
```

这个模型非常适合：

- 父进程负责调度
- 子进程负责执行独立程序
- 父进程最后回收子进程，避免僵尸进程

## 五、两种常见方案：子进程直接执行函数 vs 子进程 `exec` 新程序

在实际工程里，`fork()` 之后子进程常见有两条路：

### 方案 A：子进程继续执行当前程序中的某个函数

也就是：

- `fork()` 之后
- 不调用 `exec()`
- 子进程直接运行本程序已有逻辑

示例：

```c
void do_something() {
    printf("child is working\n");
}

int main() {
    pid_t pid = fork();
    if (pid < 0) {
        perror("fork");
        return 1;
    }

    if (pid == 0) {
        do_something();
        _exit(0);
    } else {
        waitpid(pid, NULL, 0);
    }
    return 0;
}
```

这时子进程仍然是**当前程序**，只是走到了另一条执行路径。

---

### 方案 B：子进程调用 `exec()` 执行另一个程序

示例：

```c
int main() {
    pid_t pid = fork();
    if (pid < 0) {
        perror("fork");
        return 1;
    }

    if (pid == 0) {
        execl("/bin/ls", "ls", "-l", NULL);
        perror("execl");
        _exit(1);
    } else {
        waitpid(pid, NULL, 0);
    }
    return 0;
}
```

这时子进程在 `exec()` 成功后已经不再是原程序，而是变成了 `/bin/ls`。

## 六、这两种方案本质区别是什么？

这是最值得真正想清楚的一部分。

| 对比项              | 子进程直接执行函数            | 子进程 `exec` 新程序               |
| ---------------- | -------------------- | ---------------------------- |
| 是否还是原程序          | 是                    | 否                            |
| 地址空间是否保留         | 保留 `fork` 时复制出的那份上下文 | 被新程序替换                       |
| 是否能直接使用父进程已有内存状态 | 可以使用 `fork` 时继承来的那份  | 不可以直接依赖原来的内存                 |
| 执行边界             | 与主程序强耦合              | 边界清晰、隔离更强                    |
| 适合执行外部命令         | 不适合                  | 非常适合                         |
| 成功后是否返回原逻辑       | 通常会继续运行或函数返回         | `exec` 成功后不会返回               |
| 状态传递方式           | 直接继承进程上下文            | 通过参数、环境变量、文件、管道、socket、共享内存等 |

## 七、方案 A：子进程直接执行函数

### 适用场景

### 1. 同一个程序内部的任务拆分

比如：

- 主进程继续提供服务
- 子进程负责处理一个独立计算任务
- 父子都运行同一套代码体系

### 2. 需要复用父进程已初始化好的上下文

例如父进程已经完成：

- 配置加载
- 大量只读数据初始化
- 某些缓存准备

`fork()` 后子进程可以直接继承这些状态，配合 COW，成本可能很低。

### 3. prefork / worker 模型

很多服务端程序会先完成初始化，然后 `fork()` 出多个 worker 进程去处理请求。

这类模型的好处是：

- 初始化只做一次
- 子进程共享初始化成果的内存快照（只读场景特别合适）

### 优点

### 1. 能直接利用 `fork()` 继承的上下文

这是最大的优势。

如果父进程里已经有：

- 大型配置对象
- 初始化好的查找表
- 预热后的只读缓存

子进程可以直接用，不需要重新加载。

### 2. 启动路径更短

不需要重新装载一个新程序映像。

### 3. COW 机制下可能非常高效

只要大量内存是只读或基本不改写，父子共享物理页的收益会很明显。

### 缺点

### 1. 与主程序耦合更强

子进程执行的是当前程序内部逻辑，边界不够清晰。

### 2. 继承状态太多，容易“带病下去”

例如：

- 锁状态
- 缓冲区状态
- 文件描述符状态
- 隐式全局变量

这些状态有时会让子进程行为变复杂。

### 3. 在多线程程序里更容易踩坑

如果父进程是多线程，`fork()` 后子进程只会保留调用 `fork()` 的那个线程，但地址空间里很多状态可能来自多线程环境，容易导致锁不一致、库状态异常等问题。

因此，多线程程序里常见建议是：

> **fork 之后尽快 exec，不要在子进程里做太多复杂逻辑。**

## 八、方案 B：`fork + exec` 执行另一个程序

### 适用场景

### 1. 启动外部命令

这是最典型场景，比如：

- shell 执行 `ls`
- 程序调用 `tar`
- 服务调用 `ffmpeg`

### 2. 需要清晰隔离边界

如果你希望子任务：

- 独立代码库
- 独立部署
- 独立版本演进
- 独立语言实现

那 `fork + exec` 非常合适。

### 3. 多语言组件协作

主程序可以是 C/C++，子程序可以是 Python、Go、Rust 等。

### 4. 更像“任务调度器/进程管理器”模型

父进程负责：

- 启动
- 监控
- 回收

子进程负责：

- 执行独立程序

这个职责分界非常清晰。

### 优点

### 1. 进程边界清晰

子进程 `exec` 后，已经是另一个程序，职责天然独立。

### 2. 隔离性更强

它不会继续依赖父进程的大量内部状态。

### 3. 更适合工程化拆分

如果你要：

- 调外部工具
- 做独立 worker
- 用不同语言实现模块
- 做插件式架构

那么 `fork + exec` 通常更自然。

### 4. 在复杂进程环境中更稳

尤其在多线程程序里，`fork` 后尽快 `exec` 是更常见、更稳妥的实践。

### 缺点

### 1. 不能直接复用父进程中的内存对象

`exec()` 一旦成功，原来的地址空间就没了。

如果要传状态，只能通过：

- 命令行参数
- 环境变量
- 文件
- 管道
- socket
- 共享内存
- 数据库 / RPC

### 2. 装载新程序有额外成本

相比“直接执行已有函数”，`exec` 要重新装载可执行文件、重新初始化运行时。

### 3. 交互方式更显式，也更繁琐

但这件事其实有两面性：

- 从开发便利看，是成本
- 从架构清晰看，是优势

## 九、如何选择？

如果你在工程里要做设计，可以用下面这套判断方式。

### 选“子进程直接执行函数”时，通常意味着：

- 子任务本来就是当前程序的一部分
- 你想复用父进程初始化好的只读状态
- 你更看重启动成本和上下文继承
- 你能控制好多进程继承状态带来的复杂性

### 选“`fork + exec`”时，通常意味着：

- 你要运行外部命令或另一个独立程序
- 你想要清晰的进程边界
- 你希望模块解耦
- 你在多线程程序中想降低 `fork` 后复杂逻辑的风险
- 你接受通过 IPC/参数来传递状态

一句话概括：

> **偏“内部并行任务”时，倾向子进程直接执行函数；偏“独立程序调度”时，倾向 `fork + exec`。**
